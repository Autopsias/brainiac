1 · What it is, in one paragraph

Brainiac is a local-first personal knowledge engine: a folder of plain Markdown files (the “vault”) plus a single Python command-line program (brain) that indexes those files and answers questions over them — lexically (full-text search) and semantically (vector embeddings computed on-device). AI assistants (Claude Code, Claude Desktop, Codex, Gemini CLI) use it by running the CLI in a shell. There is no server, no daemon, no cloud component, and no API key held by the system. Everything lives on the user's machine; the only network egress in the whole design is the AI assistant's own model call, which belongs to the assistant vendor, not to Brainiac.

2 · Design principles (the “why” behind every component)

  1. Files are the single source of truth. Every note is a Markdown file with a YAML header (“frontmatter”) carrying its id, type, dates, and a mandatory classification label. The search index is a derived cache: it can be deleted and rebuilt from the files at any time (brain rebuild). No state exists only in a database.
  2. Local by default. Indexing, embedding, and retrieval all run on-device (SQLite + ONNX models). No document content leaves the machine through Brainiac itself.
  3. Egress is the security budget. The load-bearing control is not disk encryption but the egress gate: what the engine is willing to print to an AI model, filtered by each note's classification tier.
  4. Two trust contexts, split capabilities. A trusted host process can write and sign; an untrusted sandbox (Claude Desktop's Linux VM, “Cowork”) can only read a published snapshot and stage drafts. The sandbox can never mutate the canonical data.
  5. Minimal persistence footprint. Exactly two OS-scheduled tasks exist (hourly maintenance + weekly synthesis). No background daemons.

4 · The data model (what a “note” is)

Two zones with different rules:

  • raw/ — immutable capture. A source document lands once, gets a sha256 of its body written into its header, and is never edited again. Original binaries are archived alongside. This gives every derived claim a provenance anchor.
  • brain/ — curated knowledge. Atomic notes (one idea each), densely linked with [[wikilinks]], organized only by a light PARA folder scheme (Projects/Areas/Resources/Archive). Structure comes from links, not folders or tags.

Every note's YAML header carries: id, type (note/decision/person/… — a small typed vocabulary), classification (one of five ordered tiers — mandatory; missing = treated as most-sensitive), dates, and optional bitemporal versioning keys: when a claim changes, the old note is superseded (chained to its successor, never deleted), so “what did we believe on date X” stays answerable. The validator enforces chain sanity: no cycles, no forks, one latest version per chain.

5 · Data flows

5.1 Retrieval (read path)

question → brain grep / bases-query      (cheap, lexical, no embedding)
         → brain search / hybrid-search  (BM25 + dense vectors fused; embeds
                                          the QUERY only, on-device)
         → brain graph-expand / get      (follow links, read full notes)
         → [egress gate]                 (classification filter — final stage
                                          before ANY result reaches stdout)

The model orchestrates these composable probes; nothing is a black box. The dossier verb bundles the common decision-state sweep (current decisions separated from raw sources, with version tensions flagged).

5.2 Capture (write path — host)

file lands in vault/inbox/  →  hourly `brain maintain` (or manual `brain sync`)
  → validate + classify + sha256 → promote to raw/ (immutable)
  → Ed25519-sign audit entry → commit to index → republish snapshot

5.3 Capture (write path — untrusted sandbox)

[VM]  brain draft-capture → .brain/capture-inbox/<id>.md   (status: draft,
                                                            trust: untrusted)
[HOST] next sync drains it: validate → sign → index → snapshot republish
[VM]  the note is now retrievable — from the new snapshot only

The VM never signs, never writes the index, never sees a key. A draft is invisible to search until the host has committed it. Fails closed: no signing key ⇒ the draft stays a draft.

5.4 Maintenance (the two scheduled tasks)

  • brain-nightly (hourly): drains the inbox, syncs the index, republishes the snapshot, and runs date-gated folds — Monday health metrics, Tuesday integrity scan, Sunday digest/curation/promotion scan + a golden-set retrieval regression probe, monthly drift-gated graph rebuild. All metadata work; it never writes prose.
  • brain-synthesis (weekly): the one model-backed session, keeping human-readable synthesis (index map, state-of-play notes) current — because prose needs a model and the engine deliberately does not hold one.

Self-observability: every maintain run appends a metrics record to health-history.jsonl (trend detection), degradations raise a deduplicated desktop notification, and an independent “watchdog of last resort” check detects that the scheduler itself died (heartbeat staleness).

6 · Trust contexts (the load-bearing split)

VisibilityEDR-visible, audit-loggedEDR-blind, ephemeral
Readsfull index (live)published read-only snapshot ONLY
Writeseverything (sign + index + snapshot)draft-capture staging only
Keysholds the Ed25519 audit key (OS secret store)can never resolve a key
Default egress capfull vaultInternal tier and below

With --role vm (or BRAIN_ROLE=vm) the write verbs are refused before any engine construction, the index opens read-only from the snapshot, and key-resolution paths are unreachable — pinned by integration tests. Note the honest limit: --role is caller-selected and defaults to host, so the split is a deployment boundary (the untrusted leg must be a separate context given only a projected vault + a capped binary), not something a single shared binary enforces by itself. See the security doc §2.2/§6.3.

7 · Installation & integration surfaces

  • PyPI: uv tool install brainiac-cli / pipx install brainiac-cli — the primary channel; a wheel-complete package (console scripts brain, brain-mcp).
  • Bootstrap installers: install.sh (macOS/Linux) and install.ps1 (Windows), also reachable via a zero-dependency npx brainiac-install. They chain uv → pipx → pip (first success wins), build the first index with the offline hash embedder (no model download at install time), and defer the real embedding model to an explicit brain warmup.
  • Client wiring: brain connect <client> writes the per-client config (claude-code, claude-desktop, codex, gemini) pointing it at a vault. The .mcpb bundle serves Claude Desktop's chat tab (the one surface that cannot run a shell) through the read-only MCP adapter — it spawns the host's brain-mcp, never vendoring a second engine.
  • Skills/plugins: curation skills ship as a Claude Code plugin marketplace in-repo (brainiac-kernel / brainiac-extras / brainiac-manager), and as zip uploads for the Cowork VM.

End-to-end sequences for a new machine / an existing machine + new project: docs/install-guide.html (plain-language) and docs/install/README.md (full detail).

8 · Technology choices (and what was deliberately avoided)

ChoiceRationale
SQLite (FTS5 + sqlite-vec)one file, zero administration, rebuildable; no client/server DB to operate or secure; brute-force fallback keeps working without the native extension
ONNX embeddings on-device (multilingual-e5-small default)no document content sent to an embedding API; works offline after a one-time ~465 MB download; deterministic hash fallback keeps the system usable with no model at all
Ed25519 + hash chain (JSONL)tamper-evident write history (each entry binds a content hash, so post-commit edits are detectable via verify-audit --check-content) at ~zero operational cost; keys live in the OS secret store
Plain Markdown + YAMLvendor-neutral, human-readable, diffable, survives any tool's demise
Python ≥3.9, short dependency listonnxruntime, tokenizers, numpy, sqlite-vec, cryptography, PyYAML, document parsers — no web framework, no ORM, no network client
Deliberately absent: no server process, no daemon, no cloud storage, no telemetry, no model API keys (the AI assistant's own vendor session is the only model call), no background sync agents beyond the two scheduled tasks.

9 · Operational properties an architect should know

  • Recoverability: the index is disposable (brain rebuild); the vault is plain files (back up like any folder); the audit chain is append-only JSONL (brain verify-audit).
  • Concurrency: single-user by design. Maintenance runs under a lock with a stale-lock auto-break; the metrics log is rotation-safe under overlap.
  • Failure posture: fail-closed everywhere it matters — unlabelled notes are treated as most-sensitive; a missing signing key blocks promotion rather than silently skipping signing; a missing embedder fails only the semantic paths (lexical search and capture keep working).
  • Observability: brain status (index stats, snapshot generation/age, pending drafts, embedder ready/pending), brain doctor, hourly health-history.jsonl, desktop degradation alarms, weekly golden-set retrieval regression score.
