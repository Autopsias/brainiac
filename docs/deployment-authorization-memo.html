Deployment Authorization — Brainiac on company laptops

Prepared for: IT Architecture & Information Security. Subject: whether to authorize Brainiac (local knowledge-engine CLI) for endpoint deployment, and under what conditions. Basis: two independent security reviews (in-house + cross-family, Codex gpt-5.6-sol) assessed at v0.16.0 (2026-07), each verified against source, plus a hardening pass that closed several findings. Current release: v0.19.10 — changes since the assessed version are in CHANGELOG.md; re-confirm the DONE conditions in §3 against the version you deploy before signing. This memo records the recommendation; the named approvers below make the decision.

Recommendation: CONDITIONALLY AUTHORIZE

Authorize deployment on single-user, company-managed laptops as a personal knowledge engine over content the user is already entitled to see — subject to the conditions in §3. Do not authorize reliance on the classification tiers as an access-control boundary between a user and data on their own machine, the sandbox/Cowork VM leg, or vaults holding regulated data (MNPI/PII/PCI) without the separate compliance sign-off in §4.

1 · Why this is a conditional yes

Brainiac is a local-first CLI: no server, no listening ports, no telemetry, no stored third-party credentials, and no network traffic of its own beyond an explicit, user-triggered model download. The security design correctly treats the AI model as the exfiltration channel and puts a deny-by-default egress gate in front of it. Both reviews independently reached the same verdict. The gaps are of two kinds: (a) engineering hardening, most of which we have now closed (§3, "Done"), and (b) policy/deployment lines the organization must draw — no code change resolves these (§3, "Org owns" and §4).

One structural fact to internalize: the vault is plaintext Markdown in user space. The classification gate governs what an AI model is shown; it is not a barrier against the laptop's own user, who can read their files directly. That is the normal trust model of any local file — acceptable — but it makes the authorization contingent on the rule in §4.1.

2 · Scope of this authorization

✓ Authorized

  • Single-user, MDM-managed laptops (macOS / Windows).
  • Vaults holding only data the user is entitled to, up to Confidential.
  • Host-role CLI use by approved AI assistants (see §4.5).
  • The read-only MCP surface, capped per §3.

✗ Not authorized (this memo)

  • Tiers used as an intra-machine access boundary.
  • The Cowork / sandbox-VM leg (EDR-blind).
  • Vaults containing MNPI / regulated data → §4 compliance gate.
  • Multi-user or shared machines.
  • Self-service install/update on managed endpoints (use §3.6 managed build).

3 · Conditions of authorization

Status legend: DONE shipped & tested as of the assessed version (v0.16.0; still present in v0.19.10) · ORG the organization's action before rollout · GATE a hard blocker if unmet.

#ConditionStatus
3.1Vault entitlement rule. Written policy: a vault may contain only data its owner is cleared to access. Classification tiers are a safety default, not a boundary. ORG GATE
3.2Full-disk encryption + owner-only ACLs on every endpoint (FileVault / BitLocker). Live-store confidentiality rests on this — app-layer encryption today covers backups only.ORG GATE
3.3Hash-pinned dependencies + CVE gate + SBOM. Install resolves the pinned lock from an approved internal index; CI audits it weekly. DONE requirements.lock, supply-chain.yml · ORG internal mirror
3.4No runtime model download. Pre-stage a reviewed, checksummed model bundle; the default download is now revision-pinned for reproducibility. DONE revision pin + $BRAIN_MODEL_CACHE · ORG sign the bundle
3.5Audit-chain integrity. Every signed write binds a content hash; verify-audit --check-content detects post-commit edits. Keys in the OS keystore, fail-closed.DONE
3.6Managed lockdown. Deploy with BRAIN_MANAGED=1: disables self-update and ad-hoc env/shell key custody (keystore-only). Updates pushed centrally. DONE · ORG set the flag via MDM
3.7Destructive-path + symlink hardening. project --dest cannot delete the vault/home/ancestors; inbox ingestion rejects symlinks (no read outside the vault). DONE
3.8Cap the MCP surface. Set BRAIN_MAX_EGRESS_TIER via managed config the AI client cannot edit (default ships full-vault). ORG
3.9Prohibit the sandbox-VM leg in this authorization (EDR-blind; optional and severable).ORG
3.10Parser sandboxing (defense-in-depth). The five document parsers are the main third-party attack surface; run ingestion in a low-privilege worker where feasible. ORG recommended, not blocking

4 · Decisions the organization must record (no code resolves these)

4.1Entitlement. Confirm the vault-entitlement rule (3.1) is owned by a named policy and enforced by onboarding, not by the tool.
4.2Model-vendor terms. The AI assistant's own model call is the real egress; authorization is contingent on the enterprise no-train / zero-data-retention agreement with that vendor. Confirm coverage for the content tiers in scope.
4.3Harness tool policy. No approved AI profile may hold vault access and an outbound web/search or arbitrary-shell tool for Confidential+ vaults — this is the surviving prompt-injection exfiltration path, and it is configured outside Brainiac.
4.4Support posture. Single maintainer, no SLA, latest-tag-only fixes (per SECURITY.md). Accept, or operate an internally-reviewed fork/mirror.
4.5Regulated data. If any vault will hold MNPI/PII/PCI, route to Legal/Compliance for DLP, retention, and e-discovery over plaintext files — out of scope for this memo.

5 · Residual risks accepted at sign-off

  • Prompt injection is mitigated, not eliminated — bounded by the gate + §4.3 harness policy; a persuasive injected instruction to an over-privileged host agent remains the residual path.
  • The trusted-host default surfaces the full vault to any process running as the user — an intentional owner decision; the compensating control is OS-level (who runs as the user) + the signed audit chain, not the gate.
  • Elevation is a CLI flag, not an enforced human gate; real containment for any untrusted leg is a filtered projection, which §3.9 defers.
  • Availability is best-effort (watchdogs, no HA) — acceptable for a single-user tool.

These are documented in full in docs/security-overview.html §6, which was corrected in this review cycle to remove earlier overstatements.

6 · Recommendation & sign-off

With conditions 3.1, 3.2 met (GATE) and 3.3–3.9 applied, the residual risk is consistent with a standard local productivity tool over entitled data. Recommended: authorize the scope in §2 for a bounded managed pilot, then review.
