Managed Deployment Runbook

Turns the “ORG” conditions of the deployment-authorization-memo.html into concrete, copy-paste MDM steps for company-managed laptops. Everything here is defensive endpoint hardening of your own machines — locking down install/update, key custody, and egress. Applies to v0.16.0+. Steps that need infrastructure you own (an internal package index, a code-signing certificate) are marked ORG-INFRA; the commands are provided, the infrastructure is yours.

Read first — the model this hardens: Brainiac's controls run in user space over plaintext files. This runbook does not make the vault a secrets vault; it makes the endpoint a managed, non-self-modifying, keystore-only, capped install. Pair it with the authorization memo's GATE conditions (FDE + the entitlement policy), which are what actually bound the blast radius.

0 · Prerequisites you provide once (ORG-INFRA)

  • An approved internal package index (Artifactory / Nexus / Azure Artifacts) mirroring PyPI, so endpoints never resolve packages from the public internet.
  • A reviewed, internally-built brainiac-cli wheel, code-signed per your policy, published to that index (build: python -m build from a reviewed tag).
  • A staged model bundle — the reviewed, checksummed multilingual-e5-small ONNX + tokenizer (see §3), distributed via MDM.
  • MDM tooling (Jamf / Intune) to push env vars, files, and the scheduled task, and to enforce FDE.

Maps to memo conditions 3.3 / 3.4.

1 · Full-disk encryption + owner-only ACLs (memo 3.2 — GATE)

The live vault, index, snapshot, and audit log are plaintext on disk; their at-rest confidentiality is FDE. Enforce it by MDM policy, not per-user opt-in.

  • macOS: Jamf → FileVault enforced, institutional recovery key escrowed.
  • Windows: Intune → BitLocker, TPM-backed, recovery key escrowed to AAD.
  • Vault directory permissions owner-only (macOS chmod 700 ~/brain; Windows: remove inherited ACLs, grant the user only).

2 · Install the engine from the pinned lock (memo 3.3)

Do not use install.sh / npx / self-update on managed endpoints. Install the internally-built wheel with dependencies pinned + hash-enforced from requirements.lock, all from your internal index:

# point pip/uv at the internal index (example — set via MDM env, not per-user)
export PIP_INDEX_URL=https://pkgs.corp.example/pypi/simple
export PIP_NO_INDEX=            # allow ONLY the internal index; no public fallback

# 1) install the exact, hash-verified dependency closure
python -m pip install --require-hashes -r requirements.lock

# 2) install the reviewed, signed engine wheel WITHOUT re-resolving deps
python -m pip install --no-deps brainiac_cli-0.16.0-py3-none-any.whl

brain --version        # expect: brain 0.16.0
Why two steps: --require-hashes -r requirements.lock guarantees every dependency matches a reviewed SHA-256; --no-deps <wheel> installs the engine itself without letting pip pull anything unpinned. The wheel's integrity comes from your code-signing + internal-index review, not a public hash.

ORG-INFRA: CI already fails a PR if requirements.lock drifts from pyproject.toml and audits it weekly (.github/workflows/supply-chain.yml). Mirror that gate in your internal pipeline before publishing the wheel.

3 · Stage the embedding model offline (memo 3.4)

Prevent any runtime Hugging Face download. Stage the reviewed model once, distribute it by MDM, and point every install at it. The default download is revision-pinned, so what you stage is reproducible.

# --- build the bundle ONCE on a reviewer's box (has network) ---
# --revision defaults to the runtime-pinned commit, so the bundle is reproducible.
python packaging/stage_model.py \
    --repo Xenova/multilingual-e5-small \
    --out ./e5-small-bundle \
    --patterns onnx/model.onnx tokenizer.json tokenizer_config.json special_tokens_map.json config.json
sha256sum -b ./e5-small-bundle/onnx/model.onnx > e5-small-bundle.SHA256   # record for review

# --- on each managed endpoint (MDM copies the bundle in, read-only) ---
export BRAIN_MODEL_CACHE=/opt/brainiac/models/e5-small        # points at the staged bundle
export BRAIN_EMBEDDER=auto                                     # uses the cache; no download

# air-gapped alternative (no model at all): lexical-only search
# export BRAIN_EMBEDDER=hash
Cache layout note: $BRAIN_MODEL_CACHE accepts either a flat snapshot dir (containing onnx/model.onnx + the tokenizer files, as staged above) or an HF cache root. Verify the recorded .SHA256 on each endpoint before trusting the bundle.

ORG-INFRA: review + checksum + (ideally) sign the bundle; verify the recorded SHA-256 on each endpoint before trusting it. Pin an internal mirror revision with $BRAIN_EMBED_REVISION if you re-host the model.

4 · Enable managed mode (memo 3.6)

The corporate lockdown switch. Set it in the machine environment (MDM), for both interactive shells and the scheduled task:

BRAIN_MANAGED=1

With it on, the endpoint refuses brain update (updates come from your pipeline) and ignores env/shell key custody — the OS keystore is the only key source (§5).

Critical interaction — the scheduled task's key. The default launchd/Task-Scheduler template injects the signing key as a BRAIN_AUDIT_KEY_PEM env literal in the task. Under BRAIN_MANAGED=1 that source is ignored, so the signing task must get its key from the keystore instead. When you register the maintenance task on a managed endpoint:
  • Remove the BRAIN_AUDIT_KEY_PEM entry from the generated task (macOS: delete that key/string pair from the .plist's EnvironmentVariables; Windows: don't set it on the task).
  • Add BRAIN_MANAGED=1 to the task's environment.
  • Ensure the audit key is in the keystore for the account the task runs as (§5), or the signing step fails closed (drafts stay staged, never committed unsigned — visible in brain status).
A no-PEM-in-plist task template is the right long-term shape; until a register_tasks --managed flag ships, do this edit at registration time.

5 · Provision the audit key in the OS keystore (memo 3.5/3.6)

Managed endpoints resolve the Ed25519 signing key only from the OS secret store. Use a non-exportable, ACL-bound entry (HSM/Secure-Enclave-backed where available).

# macOS — store the reviewer-generated PEM in the login keychain, ACL-bound
security add-generic-password -s profile-a-brain-audit-key -a "$USER" -w "<PEM>" -T ""
# ( -T "" grants NO other app access; the user approves brain on first use )

# Windows — Credential Manager (generic credential), or your HSM/CNG provider
#   Target: profile-a-brain-audit-key   User: %USERNAME%   Secret: <PEM>

The key signs every committed note so the write history is tamper-evident; it is never an encryption key and never leaves the keystore. Rotation runbook: SECURITY.md.

6 · Cap the MCP / egress surface (memo 3.8)

The host CLI defaults to the full vault; the LLM-facing MCP read surface defaults to Internal. For a differently capped deployment, set a hard ceiling via managed config the AI client cannot edit:

# machine env (MDM) — a hard ceiling for the MCP server
BRAIN_MAX_EGRESS_TIER=Internal        # or Confidential, per policy

# and/or a per-machine default cap for the CLI
BRAIN_DEFAULT_MAX_TIER=Internal

# when wiring Claude Desktop, bake the ceiling into the MCP stanza:
brain connect --client claude-desktop --max-tier Internal
Boundary detail: a host caller may change its per-call --max-tier; the VM caller cannot raise itself past the operator-owned $BRAIN_VM_MAX_EGRESS_TIER ceiling (default Internal). The strongest containment for an untrusted consumer is still a physically-filtered projection: brain project --dest <dir> --max-tier Internal ships only the allowed tiers.

7 · Verify — endpoint acceptance checklist (run post-deploy)

brain --version                       # expect 0.16.0 from the internal wheel
brain doctor --json                   # no FAIL lines; reports install channel
brain status                          # index + snapshot + embedder: ready|pending

# managed mode is active — self-update refused:
brain update && echo UNEXPECTED       # expect exit 1, "disabled on a managed endpoint"

# no network model pull (staged cache in use):
brain search "smoke test" --json | head        # returns results/egress, no HF download

# audit integrity, including post-commit edits:
brain verify-audit --check-content --json       # status ok / content_drift, 0 drift

# egress cap honored:
brain recent --json | grep -o '"withheld":[0-9]*'   # >0 if higher-tier notes exist

Pass criteria: engine is the internal build, brain update is refused, no outbound HF/PyPI traffic at runtime (confirm on the network egress logs), audit verifies, and the egress cap withholds above-tier notes.

8 · Update & rollback

  • Update is a re-deploy, not a self-update: publish a new reviewed wheel + regenerated lock to the internal index, then push §2 again via MDM. brain update stays disabled on the endpoint.
  • Rollback: re-run §2 pinning the previous wheel version + its lock. The vault and audit chain are untouched by an engine downgrade (the index is a rebuildable cache — brain rebuild if the on-disk index format changed).
  • Uninstall: remove the scheduled task, pip uninstall brainiac-cli, delete the keystore entry. Never deletes a vault or its notes — archive those deliberately.
