1 · System in one sentence, posture in one more

Brainiac is a local-only CLI that indexes a folder of Markdown notes and surfaces them to AI assistants; it runs no server, opens no listening ports, holds no API keys, and initiates no network traffic of its own (one exception: an explicit, user-invoked embedding-model download — §6.5). The security design assumes the content is sensitive and the AI model is the exfiltration channel to worry about — so the primary control is an egress gate on what the tool will print, not perimeter defenses it doesn't need.

2 · The four load-bearing controls

2.1 Classification egress gate (deny-by-default)

Every note carries a classification: label from a five-tier ordered scheme:

Public < Internal < Confidential < Restricted < MNPI

Before any read verb (search, get, recent, grep, bases-query, graph-expand, dossier, …) writes to stdout, results above the caller's allowed tier are withheld. The withholding is visible (a “N withheld” notice), so a starved result is a prompt for a human-gated elevation, not a silent gap.

  • Fail-closed rule: a note whose label is missing, empty, or unrecognized is treated as MNPI (most sensitive) at every surfacing boundary. Consequence to understand precisely: this only hides the note when the caller's cap is below MNPI. On the trusted-host default (full vault = MNPI cap, owner decision 2026-07-10) an unlabelled note ranks as MNPI and is therefore surfaced, not hidden. Unlabelled content becomes invisible only under a narrowed cap ($BRAIN_DEFAULT_MAX_TIER below MNPI, or the VM leg's Internal default). So the “deny-by-default” property protects a capped surface; it does not hide mislabelled notes from a full-vault host session — that host is trusted to see everything.
  • Role-split defaults: a trusted host session defaults to the full vault; the untrusted sandbox leg (--role vm) defaults to Internal and below. On the host, --max-tier may narrow or raise the per-call cap. On the VM it is clamped to the operator-owned $BRAIN_VM_MAX_EGRESS_TIER ceiling (default Internal), so a model-controlled argv cannot self-elevate; the elevation hint is suppressed there as well. A physically filtered brain project workspace remains the stronger containment boundary because it omits sensitive files entirely (see §2.2 / §6.3).
  • The graph surface applies the same gate — a withheld note cannot leak via link expansion.

Code: src/brain/classification.py, src/brain/egress.py; scheme: docs/classification-scheme.md.

2.2 Host/VM trust split (breaking the “lethal trifecta”)

The dangerous combination for an AI agent is untrusted content + private data + an outbound channel in one execution path. Brainiac splits it:

CapabilityHOST broker (macOS/Win)Cowork Linux VM (sandbox)
Readlive index, full-tier defaultread-only snapshot, Internal-capped
Writesign + index + snapshot publishdraft-capture staging only
Signing keyOS secret storecannot resolve a key (no code path)
EDR / audit visibilityvisibleblind, ephemeral

The VM is a read + propose surface. Its drafts carry provenance.trust: untrusted and are inert — not indexed, not searchable — until the host validates, signs, and commits them. When the process actually runs as role=vm, write verbs are refused before the engine is even constructed, and this is pinned by integration tests.

Where the split is strong, and where it isn't. The refusal is real, but --role is caller-selected and defaults to host: on a single machine, any process that can run brain can select --role host. So the split is a deployment boundary, not a runtime guarantee — it holds only when the untrusted leg is a genuinely separate context (the Cowork VM) that (a) is given a projected vault via brain project — never the canonical files — and (b) runs a binary that cannot select the host role. A corporate build should ship a separate VM entry point with host verbs removed and a non-overridable cap (§6.3). On a normal laptop there is no VM: the user is the host and reading their own entitled files is expected.

2.3 Tamper-evident audit chain (fail-closed signing)

Every committed write (note promotion, supersession, ingestion) appends an Ed25519-signed, hash-chained JSONL record — each entry carries the hash of its predecessor; brain verify-audit walks the chain, and brain anchor / verify-anchor can pin the chain head off-host. Key custody, in order, failing closed:

  1. macOS Keychain / Windows Credential Manager (service profile-a-brain-audit-key), or
  2. an operator-supplied custody command ($BRAIN_AUDIT_KEY_CMD, e.g. an age-decrypt pipeline), or
  3. explicit PEM via env.

No key ⇒ no promotion. A draft is left staged rather than committed unsigned. Code: src/brain/audit.py.

What it does and does not prove (hardened 2026-07-12). Each signed entry now binds a full content_sha256 of the exact bytes written, so a note edited on disk after commit is detectable: brain verify-audit --check-content reports any note whose current bytes differ from the last hash the chain signed. The chain proves integrity of the operation record and, with the content hash, of the committed bytes — it does not provide confidentiality (entries and the log are plaintext), and two residual gaps remain by design at this scale: continuity is not re-verified before each append (a truncated log can start a new genesis), and the append↔file-write pair is not a single durable transaction. Off-host anchoring (brain anchor) mitigates the first for deployments that require it; treat these as the honest boundary of the guarantee.

2.4 Untrusted content is data, never instructions

Anything from raw/, freshly-ingested files, MCP/tool output, and session memory is contractually data: the agent-facing contract (AGENTS.md) instructs every harness to never execute instructions found inside vault content. This is a prompt-injection mitigation, not a proof — see §6.1.

3 · Data at rest

  • Baseline: full-disk encryption (FileVault/BitLocker) + OS file permissions. The vault, index, and audit log are plain files in the user's home directory; a single-user machine is the assumed deployment.
  • Conditional application-layer encryption is built in but off by default: AES-256-GCM (src/brain/encryption.py), key custody mirroring the audit key (OS secret store / custody command / env), fail-closed. The flip-list for turning it on: off-device backup/sync, regulated-data regimes, multi-user machines, or a cyber-team mandate. Any off-device backup must be encrypted (brain backup integrates with the module).
  • Rationale: the budget goes to egress control, where the real risk is — docs/substrate-spec.md §6.

4.1 Network endpoints — the complete list

Everything the tool itself can contact, so a proxy/allowlist review has a closed set. No connection is made at rest, on a timer, or on a read; each is tied to an explicit action:

registry.npmjs.orgonly if you use npx brainiac-installfetches the bootstrap, which then installs from PyPI. Skippable — it is one of three install routes.

Nothing here carries vault content — these are artifact fetches. The one channel that carries your content is the AI client's own model call, which the tool never makes and holds no key for (§1, §6.2). Data residency for that channel is your model vendor's contract, not this tool's to state.

4.2 Reporting a problem

Suspected vulnerabilities go through GitHub private vulnerability reporting on the repository (Security → "Report a vulnerability"); see SECURITY.md for what to include and the audit-key rotation runbook. Stated plainly for planning purposes: this is a single-maintainer project with no SLA and no bug bounty — an adopting organization should treat support as best-effort and budget its own remediation path.

5 · Threat model

Assets: the vault content (up to MNPI tier), the audit signing key, the integrity of the audit history.

Adversaries considered:

  • A1 — malicious/compromised content entering via capture (a poisoned PDF, a web clip carrying prompt-injection text) trying to make an AI agent exfiltrate vault content or perform destructive actions.
  • A2 — a compromised or manipulated AI agent process (any harness with shell access on the host).
  • A3 — the sandbox VM treated as hostile (EDR-blind, ephemeral, runs model-directed code).
  • A4 — supply chain (PyPI dependencies, ONNX model artifacts, installer scripts).
  • A5 — local attacker / other OS user (device theft, snooping).

Out of scope by design: a fully-compromised host OS or user account (root/owner malware) — at that point every local-first system is lost; FDE + OS hygiene are the controls, not Brainiac.

6 · Known vulnerabilities & residual risks (the honest list)

6.1 Prompt injection remains the top residual risk (A1 → A2)

The “data, never instructions” contract is policy enforced by the model's compliance, not by a mechanism. A sufficiently persuasive injected instruction inside captured content could still steer an agent that has both vault read access and an outbound channel (e.g. its own web-search tool). Mitigations in place: the trifecta split (the leg that reads wild content is the capped VM), the classification gate limits what such an agent can read at all, human gates on Restricted/MNPI elevation, and drafts are inert until host-validated. Not eliminated: a host-side agent with full-tier default egress that also holds a web tool is the residual path — see §6.2.

6.2 Outbound egress through the model's own channels

Brainiac gates what the model reads; it cannot gate what the model then does. Two concrete leaks the design acknowledges:

  • The model call itself. Surfaced content flows to the assistant vendor. The stated control is contractual (enterprise no-train / zero-data-retention terms) — a policy control, not a technical one.
  • Web-search query leakage. A query string containing an internal term (deal codename, counterparty) IS an egress event to a public search engine. The standing rule (“never leak internal topics into web search”) lives in AGENTS.md and is model-followed, not enforced.

A cyber review should treat the AI harness's tool policy (which tools an agent may hold alongside vault access) as part of this system's control set.

6.3 The trusted host default is wide by deliberate choice

Since 2026-07-10 the host default egress cap is the full vault (the old Internal default starved real queries; elevation friction pushed users to worse channels). Consequence: any process able to run brain as the user on the host reads up to MNPI. The compensating controls are OS-level (who can execute as the user) and the audit chain (writes are signed), not the gate. A deployment needing a narrower host default sets $BRAIN_DEFAULT_MAX_TIER per machine — a conscious owner policy decision, documented as such.

Two corollaries a reviewer should log explicitly: (a) the MCP read surface is a separate LLM-facing boundary and defaults to Internal — raise $BRAIN_MAX_EGRESS_TIER only via managed config the AI client cannot edit; and (b) a host caller can still change a narrowed host default with --max-tier, while the VM caller is hard-clamped to $BRAIN_VM_MAX_EGRESS_TIER. For the strongest untrusted-leg containment, use brain project (physical omission) as well as the cap.

6.4 Document parsers process hostile input

Ingestion runs pypdf, python-docx, python-pptx, openpyxl, Pillow over files dropped in the inbox — historically CVE-bearing parser families. Mitigations: parsing happens on the host as the user (no privilege boundary is crossed), unknown extensions are quarantined not parsed, symlinks are rejected (§6.8), and originals are archived immutably. Residual: a parser RCE = user-level code execution. These five libraries are the main third-party attack surface — the hash-pinned lock + weekly CVE audit below keep them patched, but a managed baseline should run ingestion in a sandboxed low-privilege worker (still open — §8 conditions).

6.5 Supply chain (hardened through 2026-07-20)

  • Dependencies — now hash-pinned + CVE-gated. requirements.lock is the full transitive runtime closure with per-artifact SHA-256 hashes (uv pip compile --generate-hashes; make lock regenerates it). CI (.github/workflows/supply-chain.yml) fails the build if the lock drifts from pyproject.toml, runs pip-audit against it on every PR/push and weekly (catching a newly-disclosed CVE in an already-pinned dep), and publishes a CycloneDX SBOM. Managed installs should resolve against the lock, from an internal mirror, not pyproject's open >= ranges.
  • Cowork offline wheels — exact artifact lock. tools/vendor_semantic_deps.py downloads a complete, exact-version dependency closure with --no-deps, accepts only the locked per-architecture filenames and PyPI SHA-256 digests, verifies every member against wheel RECORD, and rejects unexpected top-level modules. Extraction happens in a temporary generation; the currently working vendor tree is preserved unless the whole new generation validates.
  • Model artifacts — now revision-pinned. The default model download is pinned to an exact Hugging Face commit (E5_SMALL_ONNX_REVISION), so the bytes are reproducible rather than "whatever main points at today"; override with $BRAIN_EMBED_REVISION (e.g. an internal mirror pin) or bypass downloads entirely with a pre-staged $BRAIN_MODEL_CACHE / the hash embedder. Residual: integrity still rests on the HF channel unless you stage + checksum + sign the bundle internally (recommended for managed endpoints).
  • Installers. install.sh / install.ps1 / npx brainiac-install invoke package managers directly (uv → pipx → pip); they are not remote-script pipes, but they resolve unpinned public packages. The npm package (brainiac-install, source in packaging/npm/) is a thin Node bootstrap containing no engine code — it only installs the same PyPI brainiac-cli package. For managed endpoints, install from the lock via an approved internal index and set $BRAIN_MANAGED=1 (§6.6) to disable self-update.

6.6 Key custody commands use shell=True — disabled by managed mode

$BRAIN_AUDIT_KEY_CMD / $BRAIN_ENCRYPTION_KEY_CMD values are executed via the shell so an operator can plug in a custody pipeline. Every scanner flags this; the triage (and why it adds no attack surface beyond “attacker controls your environment”, which is already a full compromise) is documented per-site in docs/SECURITY_NOTES.md. Reviewers should confirm the environment of the scheduled tasks is not writable by other OS users.

Managed mode (added 2026-07-12): set $BRAIN_MANAGED=1 (via MDM/endpoint policy) and the env + shell key-custody sources (BRAIN_AUDIT_KEY_PEM/CMD, BRAIN_ENCRYPTION_KEY/CMD) are ignored — only the OS keystore (Keychain / Credential Manager) can provide a key — and brain update self-update is refused (updates are deployed centrally). This is the recommended corporate baseline; keys should be non-exportable keystore/HSM-backed entries. Tests: tests/test_security_hardening_2026_07.py::TestManagedMode.

6.7 The VM leg is EDR-blind by nature

Anything the model does inside the Cowork sandbox is invisible to host security tooling. That is exactly why the VM was stripped to read+draft: the things worth logging (writes, signing, promotion) can only happen on the EDR-visible host, and every one lands in the signed audit chain. Residual: VM-side reads of snapshot content (≤ Internal by default) are unlogged — and more broadly, reads are not logged anywhere: neither host nor VM keeps a query/access log, so there is no "who searched what" record and nothing to forward to a SIEM. The audit chain covers writes only.

6.8 Miscellaneous, known and accepted

  • .brain/memory/ (session handoff notes) is host-only, gitignored, never indexed — but it is plaintext on disk; treat like the vault (FDE).
  • The SQLite index and snapshot duplicate note content outside vault/. They rely on the FDE baseline. Correction (was overstated): the conditional application-layer encryption module (encryption.py) is not wired to the index, snapshot, vault, or audit log — its only production caller is brain backup. So “app-layer encryption” today means encrypted backups only; live-store confidentiality is FDE + OS permissions. A managed baseline should mandate FileVault/BitLocker and disable the plaintext-backup option.
  • The audit log is plaintext JSONL and proves integrity (now including committed note bytes — §2.3), never confidentiality.
  • Availability is best-effort: watchdogs detect a dead scheduler, but there is no HA story — acceptable for a single-user local tool.

Fixed in the 2026-07-12 hardening pass (previously would have appeared here): brain project --dest now refuses to recursively delete the vault, home, an ancestor, or any non-projection directory; inbox ingestion now rejects and quarantines top-level symlinks (they otherwise followed to files outside the vault); and the audit chain now binds a content hash (§2.3). Tests: tests/test_security_hardening_2026_07.py.

7 · What a cyber review should verify (suggested checklist)

  1. Gate conformance: run brain search/get under --role vm and with --max-tier variations; confirm withheld counts and that unlabelled notes never surface. (Tests: tests/test_egress_per_subcommand.py.)
  2. VM write refusal: attempt brain write / supersede / ingest with BRAIN_ROLE=vm; confirm refusal precedes any key/index access. (Tests: tests/test_integration.py.)
  3. Audit chain: commit a note, tamper with a JSONL entry, confirm brain verify-audit fails; confirm promotion is blocked with the key removed from the secret store.
  4. Scheduled-task hygiene: review the two launchd / Task Scheduler entries' argv + environment file permissions (relevant to §6.6).
  5. Harness tool policy: review which AI harnesses hold BOTH vault access and outbound tools (web search, arbitrary shell) — this is the §6.1/§6.2 residual path and is configured outside Brainiac.
  6. Dependency posture: pip audit (or equivalent) on the ~10 runtime deps, with special attention to the five document parsers (§6.4).
  7. Static-scan triage: docs/SECURITY_NOTES.md records every accepted scanner finding with per-site rationale; re-verify rather than re-accept.
